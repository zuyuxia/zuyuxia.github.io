TCP/UDP

  1. TCP和UDP的区别

    • TCP是面向连接的传输控制协议；UDP提供的是无连接的数据报服务

    • TCP具有高可靠性，确保传输数据的正确性；UDP在传输数据时不建立连接，不对数据报进行检测修改，不需等待对方的应答，所以会出现分组丢失、重复、乱序，应用程序负责传输可靠性方面的所有工作

    • TCP对系统资源要求较多；UDP对系统资源要求较少

    • UDP有很好的实时性，工作效率比TCP高

    • UDP的段结构比TCP的段结构简单，网络开销小

  1. TCP头部的标志位

    • URG：标志紧急指针是否有效

    • ACK：标志确认号是否有效（确认报文段），解决丢包问题

    • PSH：提示接收端立即从缓冲读走数据

    • RST：表示要求对方重新建立连接（复位报文段）

    • SYN：表示请求建立一个连接（连接报文段）

    • FIN：表示关闭连接（断开报文段）

  2. 三次握手

    • first，客户机发送SYN位，SEQ_NUM = S的包到服务器（客户机处于SYN_SEND状态）

    • second，服务器发送ACK、SYN位，且ACK_NUM = S+1，SEQ_NUM = P的包到客户机（服务器处于SYN_RECV状态）

    • third，客户机发送ACK位，且ACK_NUM = P+1的包到服务器（客户机处于ESTABLISH状态，服务器处于ESTABLISH状态）

  3. 四次挥手

    • first，客户机发送FIN位，SEQ_NUM = Q的包到服务器（客户机处于FIN_WAIT_1状态）

    • second，服务器发送ACK位，且ACK_NUM = Q+1的包到服务器（服务器处于CLOSE_WAIT状态，客户机处于FIN_WAIT_2状态）

    • 客户到服务器连接断开，但服务器到客户连接还未断开，此处有等待

    • third，服务器发送FIN位，且SEQ_NUM = R的包到客户机（服务器处于LAST_ACK状态，客户机处于TIME_WAIT状态）

    • 此处有等待

    • fourth，客户机发送最后一个含有ACK位，且ACK_NUM = R+1的包到服务器（服务器处于CLOSED状态）

  4. 为什么握手是三次，挥手是四次

    • 对于握手，只需要确定双方通信时的初始化序号，保证通信不会乱序

    • 第三次握手的原因：假如第二次握手失败，那么客户机就会超时重发连接请求，而服务器对客户机已经有了一个连接。这会造成服务器对客户有多个连接，造成资源的浪费

    • 对于挥手，TCP是双工的，发送双方都需要FIN和ACK

  5. TIME_WAIT状态的作用

    • 客户端发送的ACK包可能会丢失并导致服务器重新发送FIN消息，如果不存在这个TIME_WAIT状态，那么当重传的FIN包到达时，由于客户端的TCP依据不再有连接信息，所以就用RST消息应答，导致服务器进行错误的状态而不是有序终止的状态

    • TIME_WAIT为连接中某些因为网络原因迟迟未到达的数据提供从网络中消失的时间。如果客户端和服务器在断开连接后，再一次重新建立起了连接，并且连接的端口与上一次相同。那么这些迟到的数据就可能到达新的连接，就有可能被看成是新的数据，被重新接收，就会破坏新的连接

  6. TCP的连接状态

    • CLOSED
    • LISTEN
    • SYN_SEND
    • SYN_RECV
    • ESTABLISH
    • FIN_WAIT_1
    • CLOSE_WAIT
    • FIN_WAIT_2
    • LAST_ACK
    • TIME_WAIT

  7. 超时重传

    • 发送端发送报文后若长时间未收到确认的报文则需要重发该报文

    • 可能的情况：发送的数据未到达接收端；接收端接收到数据，但ACK报文返回过程中丢失；接收端拒绝或者丢弃数据

  8. RTO（重传间隔）

    • 从上一次发送数据，到下一次重发之间的时间

    • 通常是前一次重传间隔的两倍；重传次数到达上限后停止重传

  9. RTT

    • 数据从发送到接收到对方响应之间的时间间隔

    • 这个数据往往不稳定

  10. 流量控制原理（点对点）

    • 接收方通过TCP头部中的窗口字段告知发送方本方可接收的最大数据量，用以解决发送速率过快导致接收方不能接收的问题

    • 发送窗：限制发送方可以发送的数据大小

    • 接收窗：标记可以接收的数据大小

    • 发送窗内的数据只有当接收到接收端某段发送数据的ACK响应时才移动发送窗，左边缘紧贴刚被确认的数据；接收窗只有接收到数据且最左侧连续时才移动接收窗口

  11. 拥塞控制（网络链路全局）

    • 防止数据过多注入网络中导致网络资源过载

    • TCP拥塞控制算法：慢开始 & 拥塞控制；快速重传 & 快速恢复

  12. TCP的可靠数据传输如何保证

    • 建立连接

    • 序号机制

    • 数据校验

    • 超时重传

    • 窗口机制

    • 拥塞控制

  13. TCP的socket交互流程

    • 服务器：socket -> bind ->listen -> accept -> read/write ->close

    • 客户端：socket -> connect -> write/read -> close

  14. UDP

    • UDP是一个非连接协议

    • UDP使用尽最大努力交付，即不保证可靠交付

    • UDP首部只要8个字节，开销小

    • UDP是面向报文的

    • UDP没有拥塞控制

    • UDP支持一对一、一对多、多对一以及多对多的交互通信

  15. TCP/UDP的场景应用

    • TCP：SMTP、HTTP、FTP、TELNET

    • UDP：即时通信、DNS、DHCP、在线视频

  16. TCP粘包怎么处理

    • 加包头，包头中包含数据包长度

    • 约定好数据包的固定长度，发送、接收固定长度的数据包

    • 数据包之间设置边界，添加特殊符号

  17. TCP三次握手对应socket中的函数情况

    • 服务器在调用socket，bind，listen完成准备工作后，就开始等待连接

    • 客户端通过connect连接服务器，这里客户端向服务器发送SYN分节，第一次握手，connect进入阻塞状态

    • 服务器监听到客户端的连接请求，收到客户端的SYN分节，调用accept接收请求，并向客户端发送SYN和ACK分节，accept进入阻塞状态

    • 客户端收到服务器端的SYN和ACK分节之后，connect函数返回，并对SYN分节进行确认；之后服务器收到ACK分节后，accept函数返回，完成三次握手，建立连接

    • 客户端的connect在三次握手中的第二次返回，服务器的accept在第三次返回
